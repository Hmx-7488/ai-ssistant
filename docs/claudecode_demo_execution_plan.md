## 二、与当前餐饮项目对齐的总方向

Claude Code 后续的工作，优先保持与我们正在推进的餐饮客服项目一致。

### 目标架构

统一采用三层处理架构：

1. **确定性查询层**
   - 菜单
   - 价格
   - 营业时间
   - 地址
   - WiFi
   - 发票
   - 停车
   - 包间
   - 是否可调辣/去香菜/打包

2. **结构化推荐层**
   - 人数
   - 预算
   - 人群（儿童、老人、孕妇、健身、素食）
   - 忌口（不吃辣、花生过敏、海鲜过敏等）
   - 场景（一人食、夜宵、下饭、家庭聚餐）

3. **LLM/RAG 兜底层**
   - 开放式复杂问题
   - FAQ 扩展问答
   - 非结构化补充表达

原则：

- 能查表就不走大模型
- 能规则处理就不走 RAG
- 只有结构化规则无法完整覆盖时再走模型

## 三、Claude Code 必须遵守的执行原则

### 1. 优先级原则

先做“稳”，再做“炫”。

优先级顺序固定为：

1. 编码/数据/启动稳定性
2. 演示必过场景
3. 多轮对话承接
4. 前端演示质感
5. 多 Agent 讲述性增强
6. 额外花哨功能

### 2. 改动约束

1. 不要假设工作区干净
2. 不要覆盖已有用户改动
3. 每做完一批必须本地验证
4. 没有证据不要宣称“已完成”
5. 任何会影响演示稳定性的改动，都必须先加回归验证

### 3. 演示优先原则

如果“更高级的架构”和“更稳定的演示效果”冲突，优先选择后者。

面试演示场景下，稳定性高于理论最优。

## Phase 3：把多轮对话做成真承接，而不是礼貌重问（P1）

### 目标

这是和当前餐饮项目后续方向最需要对齐的部分。

### 当前判断

项目里已有 `ConversationState` 和 follow-up 识别，这是很好的起点，但还不够。

### 必须补强的能力

1. 记住上一轮约束
   - 人数
   - 预算
   - 人群
   - 忌口
   - 已推荐菜品
   - 当前场景

2. 识别承接词
   - 还有别的吗
   - 换一个
   - 不要这个
   - 刚才那个
   - 改成三个人
   - 预算改成150
   - 换个清淡点的

3. 排除上轮推荐
   - 用户说“换一个”“不要这个”时，不能把同样结果又推荐回来

4. 约束增量改写
   - 例如：
     - 第一轮：`推荐适合老人吃的菜`
     - 第二轮：`再来个清淡点的`
     - 第三轮：`两个人吃`
   - 系统应合并成新的约束集合后再重算

### 必过多轮用例

1. `推荐适合老人吃的菜` -> `再来个清淡点的`
2. `推荐小孩吃的菜` -> `不要这个，换一个`
3. `3个人150元预算` -> `改成两个人`
4. `花生过敏推荐吃什么` -> `再来个主食`
5. `一个人晚上吃点什么` -> `不要炒饭`

   - 单一事实源
   - 自动化回归
   - API 健康检查
